# i18n & RTL

The template ships with [i18next](https://www.i18next.com/) +
[react-i18next](https://react.i18next.com/) wired into the provider tree,
two starter locales (`en`, `ar`), a language-switcher widget in the
navbar, and a `DirectionProvider` that keeps `<html dir>` / `<html lang>`
in sync with the active language.

If a project doesn't need translations, run
`npm run init -- --no-i18n` to strip everything in one command (see
[Opting out](#opting-out) below).

## Where things live

```
src/shared/i18n/
├── index.ts        # i18next config + helpers (getDirection, SUPPORTED_LANGUAGES)
├── en.json         # English translations (the source of truth)
└── ar.json         # Arabic translations (RTL)

src/shared/lib/direction.tsx     # DirectionProvider + useDirection() hook
src/widgets/language-switcher/   # Dropdown that calls i18n.changeLanguage()
```

Translation files are flat JSON keyed by feature (e.g. `nav.home`,
`login.title`, `dashboard.stats.activeUsers`). Keep keys in
`camelCase` and group by view or feature, not by component.

## Using translations in a component

```tsx
import { useTranslation } from 'react-i18next'

export function Hello() {
  const { t } = useTranslation()
  return <h1>{t('home.title')}</h1>
}
```

For dynamic keys driven by data, use a template literal:

```tsx
const items = ['routing', 'query', 'protected'] as const

function Capabilities() {
  const { t } = useTranslation()
  return (
    <ul>
      {items.map((key) => (
        <li key={key}>{t(`home.capabilities.${key}`)}</li>
      ))}
    </ul>
  )
}
```

## Adding a new translation key

1. Add the key to `src/shared/i18n/en.json` first — `en` is the
   fallback, so it must contain every key the app uses.
2. Mirror it in `src/shared/i18n/ar.json` (or any other locale you've
   added). If a translation is missing, i18next falls back to `en`.
3. Reference the key in components via `t('your.new.key')`.
4. Re-run the dev server — i18next picks up JSON edits via HMR.

## Adding a new language

1. Drop a `xx.json` file next to `en.json` / `ar.json`.
2. Import it in `src/shared/i18n/index.ts`, add it to the
   `resources` map, and append the code to `SUPPORTED_LANGUAGES`.
3. If the language is RTL, add it to the `RTL_LANGUAGES` set in the
   same file — `DirectionProvider` reads from that set to flip
   `<html dir>` automatically.
4. Add the human-readable name under the `languages` key in every
   existing locale's JSON so the switcher renders correctly.

## Switching language programmatically

```tsx
import { useTranslation } from 'react-i18next'

function MyButton() {
  const { i18n } = useTranslation()
  return <button onClick={() => i18n.changeLanguage('ar')}>عربي</button>
}
```

The user's choice is persisted to `localStorage` (key
`react-template:language`) by the language detector and rehydrated on
next load.

## RTL: `useDirection()`

```tsx
import { useDirection } from '@/shared/lib'

function ArrowIcon() {
  const dir = useDirection() // 'ltr' | 'rtl'
  return <ArrowRight className={dir === 'rtl' ? 'rotate-180' : ''} />
}
```

For most icons / spacing, prefer Tailwind's `rtl:` variant or the
logical properties (`ms-*`, `me-*`, `ps-*`, `pe-*`) — they flip
automatically and don't need the hook. Only reach for `useDirection()`
when you need branchy logic that Tailwind can't express (e.g. picking
between two SVG components, computing offsets in JS).

## Testing components that use `t()`

`renderWithProviders` in [src/test/test-utils.tsx](../src/test/test-utils.tsx)
already wraps the tree in `I18nextProvider`, so `useTranslation()`
"just works" inside tests. The default language is `en`. To exercise
a different locale:

```tsx
import { act, render } from '@testing-library/react'
import { i18n } from '@/shared/i18n'

await act(async () => {
  await i18n.changeLanguage('ar')
})
```

Remember to reset back to `en` in an `afterEach` so tests stay
independent.

## Opting out

`npm run init -- --name="My App" --no-i18n` removes everything in
this doc in one command. It:

- uninstalls `i18next`, `react-i18next`, and
  `i18next-browser-languagedetector` from `package.json`
- deletes `src/shared/i18n/`, `src/shared/lib/direction.tsx` (+ test),
  and `src/widgets/language-switcher/`
- unwraps `I18nextProvider` / `DirectionProvider` from
  `src/app/providers.tsx` and `src/test/test-utils.tsx`
- rewrites every `t('foo.bar')` back to the corresponding English
  string from `en.json`. The codemod is JSX-aware: `{t('foo')}` becomes
  bare text, attributes stay quoted, and dynamic template-literal calls
  (`t(\`prefix.${key}\`)`) become an inline `Record<string, string>`
  lookup so the same key-based map keeps working without i18next.
- runs `prettier --write` over the touched files so the diff is clean

After the script finishes, run `npm install` to drop the i18n packages
from `node_modules`, then commit.
